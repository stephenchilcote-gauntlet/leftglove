The Problem

Agents are fast,
but blind

Agents build fast — parallel, insightful, tireless
But context-free — no persistent model of what the app is from the user's side
Screenshots & scraping exist — but they're fuzzy, token-expensive, and have no memory of what should be there
The real goal: keep agents on the rails of designer intent and actual user value

Current State of the Art

E2E testing doesn't solve this

Playwright
Browser automation — manual locators, manual maintenance
Appium
Mobile automation — same problems, different platform
Cypress
Browser testing — tightly coupled to DOM structure
Postman
API testing — disconnected from what users see
No real inventory — no shared map of what's actually on each screen
Phantom users — tests don't distinguish guest from admin from power user
Wrong verb level — "user logs in" instead of "user clicks submit-button"
Manual & tightly coupled — page objects break every time the frontend changes

What Exists Today

ShiftLefter — vocabulary enforcer

Every subject defined — guest, admin, power-user, test-harness — no phantom users
Every object defined — username-field, submit-button, cart-icon — bound to stable locators
Verbs bound to interface level — clicks, types-into, reads — not domain abstractions
DDD ubiquitous language, enforced — the whole team, code included, uses the same terms
Reference something that isn't in the glossary? Test fails. Drift is impossible.

The Capstone

The Sieve

Point it at a running app — reads a page, classifies every element
Walks the application — expands widgets, follows links, starts over on the next page
Classifies automatically — input, clickable, readable, static text, chrome
Interface-agnostic architecture — web browser first, but the same pattern works for mobile, REST, GraphQL, SMS
Output: structured inventory with stable locators → ShiftLefter glossary

Disambiguation

The Toddler Loop

LLM reviews uncertain elements — "What's this? What does it do? What do you call it?"
You answer in your language — your team's terms, your domain's vocabulary
Community detection — elements that always appear and act together cluster into groups. "These five elements are your login form." Confirm or correct.
Minutes instead of days — complete, labeled, validated glossary

Immediate Value

What the Sieve gets you

Re-sieve after every change — element disappeared? contract flags it. New element? Toddler Loop asks about it. Locator broke? You know immediately.
Test generation — the inventory defines what actions are possible. Inputs mean typing. Buttons mean clicking. Tests fall out of the vocabulary.
Agent guardrails — Claude gets a machine-readable map of what users actually see and do, not just what the code says should happen
Claude can drive your app — with stable locators and a classified inventory, an agent can actually navigate and verify

Where This Goes

The graph of user intent

Connect inventories across pages — a graph of every path a user can take
Derive use cases from paths — sequences of actions contract into domain verbs: "logs in," "checks out"
Use cases bridge platforms — web and mobile have different interfaces, but equivalent use cases means feature parity
Role mutation — swap admin's graph to guest, verify every path is blocked
The real goal: a deterministic graph of your app's surface — user intent pinned down precisely — so agents can be freed to do what they're best at
